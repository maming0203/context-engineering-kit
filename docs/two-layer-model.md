# 两层模型：为什么 JIT 需要分层

## 为什么 JIT 需要两层

Just-In-Time 的核心是"延迟加载"——只在需要时才付出成本。但有一个前提：**某些东西必须始终存在**，否则系统无法正常运转。

这就是两层模型的由来：

- **基础设施层（L0）**：系统的"操作系统"——身份、格式、安全边界。缺了它，Agent 连自己是谁都不知道。
- **任务层（L1）**：系统的"应用程序"——领域知识、工具用法、示例。按需加载，用完即走。

如果只有一层，你要么全量加载（浪费），要么什么都不加载（无法工作）。两层分离让系统同时拥有**稳定性**和**灵活性**。

## 巨石 Skill 拆分的教训

### 真实案例：从 2000 行到 5 个 Skill

在 Agent Assembler 的早期版本中，我们有一个名为 `financial_analysis` 的巨石 Skill，长达 2000+ 行。它试图覆盖：

- 盈亏平衡分析
- 现金流预测
- 合规性检查
- 财务报表解读
- 税务计算

**问题 1：每次对话都加载 2000 行**

用户问"帮我算一下盈亏平衡点"，系统注入了全部 2000 行——其中 80% 与当前问题无关。模型在无关内容中"迷路"，回答质量下降。

**问题 2：Skill 内部耦合严重**

修改"合规性检查"的规则时，不小心破坏了"现金流预测"的逻辑。巨石 Skill 的维护成本随规模指数增长。

**问题 3：无法组合**

用户问"帮我做盈亏平衡分析，并检查是否符合监管要求"——系统只能加载整个巨石 Skill，无法精准组合。

### 拆分后的效果

我们将巨石 Skill 拆分为 5 个独立 Skill：

| Skill | 行数 | 触发场景 |
|-------|------|----------|
| `break_even_analysis` | 120 | 用户提到盈亏平衡、保本点 |
| `cashflow_forecast` | 180 | 用户提到现金流、资金预测 |
| `compliance_check` | 200 | 用户提到合规、监管、法规 |
| `financial_statement` | 150 | 用户提到报表、资产负债表 |
| `tax_calculation` | 160 | 用户提到税、纳税、税率 |

**效果**：
- 平均上下文消耗下降 70%
- 回答准确率提升 25%
- 维护成本从"改一处怕三处"降为"各管各的"

## 基础设施层 vs 任务层的边界

### 基础设施层（L0）包含

- **身份定义**：你是谁、你的角色、你的语气
- **输出格式**：JSON schema、Markdown 模板、结构化输出要求
- **安全边界**：拒绝策略、内容过滤、隐私规则
- **核心工具描述**：所有场景都可能用到的基础工具（如搜索、计算）
- **全局约束**：语言偏好、长度限制、引用要求

### 任务层（L1）包含

- **领域 Skill**：特定任务的知识与流程
- **参考文档**：API 文档、法规条文、产品手册
- **Few-shot 示例**：特定场景的输入输出示例
- **任务约束**：特定场景的额外规则（如"财务建议必须加免责声明"）

### 判断标准

问自己三个问题：

1. **如果去掉它，Agent 在所有场景下都会出错吗？** → 是 → L0
2. **如果去掉它，只在特定场景下出错？** → 是 → L1
3. **它的体积是否超过 200 行？** → 是 → 必须进 L1（L0 应精简）

## 识别规则：什么该进基础设施层

### 规则 1：高频共性

如果某段内容在 80% 以上的对话中都需要，它属于 L0。

**示例**：
- "你是一个专业的财务顾问" → 几乎每次都需要 → L0
- "盈亏平衡点的计算公式是..." → 只在财务分析时需要 → L1

### 规则 2：不可省略

如果省略后会导致 Agent 行为失控或违反安全策略，它属于 L0。

**示例**：
- "不要提供具体的投资建议" → 安全边界 → L0
- "使用表格展示对比数据" → 格式偏好 → L0（如果全局适用）

### 规则 3：体积精简

L0 的总长度应控制在 **500 token 以内**。超过这个阈值，说明你把太多东西塞进了基础设施层。

**检查方法**：
```
L0 token 数 / 总上下文窗口 < 5%
```

如果比例过高，重新审视：哪些内容其实只在特定场景需要？

### 规则 4：稳定性

L0 内容应该很少变动（月级别），L1 内容可以频繁更新（天级别）。

如果你发现某段 L0 内容每周都在改，它可能更适合放在 L1，通过配方动态注入。

## 总结

两层模型不是银弹，但它提供了一个清晰的思维框架：

- **L0 是地基**：稳固、精简、不变
- **L1 是家具**：灵活、丰富、按需摆放

当你不确定某段内容该放哪里时，回到这三个问题：高频吗？不可省略吗？精简吗？答案会自然浮现。
